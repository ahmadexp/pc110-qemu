# Disks

Boot media. These contain copyrighted IBM software (PC DOS / Personaware) and
are **not** included in this repository — supply your own.

| File | How to get it |
| --- | --- |
| `Personaware-disk.img` | Bootable disk built from a raw PC110 partition dump: `python3 tools/make-disk.py your-partition-dump.img disks/Personaware-disk.img` |
| `easysetup-floppy.img` | Generated by `boot/build-floppy.sh` from `roms/pc110_bios.bin` (Easy-Setup loader + unpacked image). |

If your Personaware dump is already a full disk image (has an MBR / boots on its
own), you can use it directly as `Personaware-disk.img` and skip `make-disk.py`.

## The "low disk space" dialog — it's the cluster size, not the size

Personaware can pop a Japanese **"注意 … ディスクの残りが少なく"** ("Caution: disk
space is running low") dialog at startup. It looks like a free-space warning, but
it is **not** about free space or disk size — it is triggered by the **FAT
cluster size** (verified across two disk sizes):

| Disk | Cluster size | FS | Free | Dialog |
| --- | --- | --- | --- | --- |
| 8 MB CF dump | 512 B | FAT16 | 4.5 MB | **yes** |
| 8 MB, reformatted | 2 KB | FAT16 | 4.6 MB | **no** |
| 4 MB, 512 B clusters | 512 B | FAT16 | 0.9 MB | **yes** |
| **4 MB, 1 KB clusters** ← shipped | 1 KB | FAT12 | 0.9 MB | **no** |

512-byte clusters trip the dialog at *both* sizes (even the 4 MB disk with only
0.9 MB free); ≥1 KB clusters clear it regardless of size or free space. The real
PC110's 4 MB internal storage is FAT12 with large clusters, so it never warns —
but when you dump it onto a CF, the CF is FAT16 with 512-byte clusters (~15,500
clusters), which Personaware's disk-space code mishandles and reports as "low
space."

The image shipped here matches the real machine: a **4 MB FAT12 volume with
1 KB clusters**. To rebuild one: `mformat … -c 2` on a 4 MB (128 cyl × 2 × 32)
partition, patch the BPB geometry back to 32 sectors/track (mtools rewrites it),
splice the IBM PC-DOS boot loader onto the fresh BPB, and copy the system files
(`IBMBIO.COM`, `IBMDOS.COM`, `COMMAND.COM`) **first** so `IBMBIO.COM` stays root
slot 0 (the boot sector requires it there).

## CONFIG.SYS tweak for QEMU

The stock PC110 `CONFIG.SYS` puts the EMS page frame at `FRAME=E000`, which
collides with SeaBIOS on QEMU and freezes the boot. After building the disk,
move the frame out of the way (via `mtools`, using the 32-sector partition
offset):

```
mcopy -i disks/Personaware-disk.img@@16384 ::CONFIG.SYS /tmp/config.sys
# edit the EMM386.EXE line, e.g.:
#   DEVICE=C:\DOS\EMM386.EXE RAM I=B000-B7FF X=DC00-DFFF X=E000-EFFF FRAME=CC00
mdel  -i disks/Personaware-disk.img@@16384 ::CONFIG.SYS
mcopy -i disks/Personaware-disk.img@@16384 /tmp/config.sys ::CONFIG.SYS
```

## Real-BIOS boot: remove EMM386

The **real-BIOS** path (`scripts/run-realbios.sh`) needs a `CONFIG.SYS` that does
**not** load EMM386. EMM386's V86/paging monitor assumes faithful protected mode
and conflicts with the PC110 driver stack's "unreal mode" segment usage; with it
loaded the boot faults out (EMM386 "error #06/#13") before the GUI. Removing it
leaves the driver's loose protected mode as the only regime and the boot runs
clean to the Personaware desktop. Make a real-BIOS variant of the disk:

```
cp disks/Personaware-disk.img disks/Personaware-realbios.img
mcopy -i disks/Personaware-realbios.img@@16384 ::CONFIG.SYS /tmp/cfg
sed -e 's/^DEVICE=C:\\DOS\\EMM386.EXE.*/REM EMM386 removed for the real-BIOS path/' \
    -e 's/^DOS=HIGH,UMB/DOS=HIGH/' -e 's/^DOSDATA=UMB/REM DOSDATA=UMB/' \
    -e 's/^DEVICEHIGH=/DEVICE=/' /tmp/cfg > /tmp/cfg.noemm
mdel  -i disks/Personaware-realbios.img@@16384 ::CONFIG.SYS
mcopy -i disks/Personaware-realbios.img@@16384 /tmp/cfg.noemm ::CONFIG.SYS
# then: PC110_DISK=disks/Personaware-realbios.img scripts/run-realbios.sh
```

(Booting the *stock* EMM386 configuration on the real BIOS is future work — it
needs a V86-aware loose-PM model.)
